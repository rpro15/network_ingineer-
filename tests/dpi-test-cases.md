# DPI Test Cases

## Цель

Проверить корректность работы DPI-сигнатур по трем направлениям:

- точность детекта на целевом трафике
- контроль ложных срабатываний на нецелевом трафике
- стабильность под нагрузкой и в граничных условиях

## Предусловия

- версия сигнатур зафиксирована
- baseline-результаты прошлой версии сохранены
- тестовый набор `pcap` доступен и размечен
- точка съема счетчиков/логов согласована

## Матрица тестов

| ID | Сценарий | Входные данные | Что проверяем | Критерий PASS | Статус |
|---|---|---|---|---|---|
| DPI-001 | Позитивный кейс: базовый поток | known-good pcap A | сигнатура срабатывает на целевой поток | hit есть, классификация верная | Planned |
| DPI-002 | Позитивный кейс: альтернативный handshake | known-good pcap B | устойчивость сигнатуры к варианту сессии | hit есть, без деградации точности | Planned |
| DPI-003 | Негативный кейс: близкий протокол | non-target pcap A | ложные срабатывания | false positive = 0 для набора | Planned |
| DPI-004 | Смешанный фон + целевой трафик | mixed replay profile | стабильность hit-rate и шума | hit-rate в норме, без всплеска FP | Planned |
| DPI-005 | Фрагментация пакетов | fragmented sample | корректность классификации на фрагментах | классификация стабильна | Planned |
| DPI-006 | Ретрансмит и потери | lossy replay profile | устойчивость к сетевым искажениям | нет дублей/ложных hit | Planned |
| DPI-007 | Вариант с TLS | encrypted sample set | поведение по спецификации сигнатуры | результат соответствует design note | Planned |
| DPI-008 | Нагрузочный прогон | high-rate replay | влияние на CPU/latency/counters | без критичной деградации | Planned |

## Порядок прогона

1. Проверить базовое состояние узла DPI и версию сигнатур.
2. Прогнать `DPI-001..DPI-003` (точность и ложные срабатывания).
3. Прогнать `DPI-004..DPI-007` (устойчивость в сложных условиях).
4. Прогнать `DPI-008` (нагрузка и ресурсные метрики).
5. Сравнить результаты с baseline и сформировать вывод по релизу.

## Минимальный набор evidence

- версия сигнатур и timestamp прогона
- счетчики hit/flow до и после теста
- выборка логов с подтверждением результата
- системные метрики для нагрузочного теста
- финальный вердикт: `go / no-go`
