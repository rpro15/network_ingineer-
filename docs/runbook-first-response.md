# Runbook: First Response to Network Incident

## Цель

За первые 15-30 минут подтвердить влияние, локализовать зону проблемы и подготовить корректную эскалацию.

## Шаг 1. Триаж (0-5 минут)

- Зафиксировать: что не работает, с какого времени, у кого, насколько критично.
- Определить приоритет инцидента.
- Сразу дать первый статус заинтересованным сторонам.

## Шаг 2. Базовые проверки (5-15 минут)

- Запуск L3 triage smoke: `scripts/run_dpi_smoke.sh --target <service-ip-or-host>`.
- Проверка доступности и деградации (`ping`, `mtr`, `ss`, интерфейсные счетчики).
- Проверка control-plane: соседства и маршруты.
- Проверка data-plane: дропы/ошибки/очереди.

## Шаг 3. Локализация и гипотеза (15-25 минут)

- Ограничить problem scope: сегмент, линк, протокол, сервис.
- Сформулировать 1-2 рабочие гипотезы.
- Подтвердить гипотезу фактами (лог/счетчик/capture + evidence из `smoke_result_*`).

## Шаг 4. Действие и проверка (25-30 минут)

- Внести минимально рискованное изменение (или подготовить эскалацию).
- Проверить восстановление сервиса.
- Зафиксировать результат и следующий шаг.

## Шаблон статуса

"Инцидент подтвержден, затронут [сервис/сегмент], текущий приоритет [P?]. Проверяем [гипотеза], следующий апдейт через [N] минут."

## Когда эскалировать немедленно

- Широкий impact на критичный бизнес-сервис.
- Признаки циклической деградации без стабильного окна.
- Риск усугубить инцидент без старшего инженера.
